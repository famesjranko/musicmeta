```kotlin
/**
 * Enriches multiple requests sequentially, emitting each result as it completes.
 *
 * Results are emitted as a cold [Flow]. Cancelling collection (e.g., via [take])
 * stops processing remaining requests cooperatively.
 *
 * Cache hits return immediately without rate-limiter delay because the
 * underlying [enrich] call short-circuits on cached data.
 */
fun enrichBatch(
    requests: List<EnrichmentRequest>,
    types: Set<EnrichmentType>,
    forceRefresh: Boolean = false,
): Flow<Pair<EnrichmentRequest, EnrichmentResults>> = flow {
    for (request in requests) {
        emit(request to enrich(request, types, forceRefresh))
    }
}
```

2. In `DefaultEnrichmentEngine.kt`:
   - Add import for `kotlinx.coroutines.flow.Flow` and `kotlinx.coroutines.flow.flow`
   - Add explicit override after the `enrich()` method (after line 116, before `invalidate()`). Same logic but allows future optimization without interface change (BATCH-04):

```kotlin
override fun enrichBatch(
    requests: List<EnrichmentRequest>,
    types: Set<EnrichmentType>,
    forceRefresh: Boolean,
): Flow<Pair<EnrichmentRequest, EnrichmentResults>> = flow {
    for (request in requests) {
        emit(request to enrich(request, types, forceRefresh))
    }
}
```

Note: The `for` loop with `emit()` is a suspension point, so Flow cancellation (e.g., `take(N)`) cooperatively stops iteration after N emissions. Cache hits in enrich() short-circuit before hitting the rate limiter, so they return immediately — satisfying BATCH-04.
