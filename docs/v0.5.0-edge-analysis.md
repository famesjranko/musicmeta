# v0.5.0 Edge Analysis Report

**Date:** 2026-03-22
**Scope:** Post-milestone edge testing of all v0.5.0 features
**Providers active:** All 11 (Last.fm, Fanart.tv, Discogs keys provided)

## Methodology

### How testing was conducted

1. **Showcase test review** — Started by updating `EnrichmentShowcaseTest.kt` (the existing E2E diagnostic) to cover v0.5.0 features. Enriched the `snippet()` output to show GenreTag confidence scores, Credit roleCategory, ReleaseEdition details, and ArtistTimeline events. Replaced the stale hardcoded coverage gaps section with a dynamic coverage matrix from `engine.getProviders()`. Added a dedicated v0.5.0 feature spotlight test (test 09).

2. **Showcase run (all providers)** — Ran the updated showcase with all API keys. This immediately surfaced the RELEASE_EDITIONS bug (NotFound for OK Computer) and showed the genre merge only activating for some artists.

3. **Bug investigation** — Used targeted debug tests to trace the RELEASE_EDITIONS failure. Created `DebugEditionsTest.kt` to isolate identity resolution → API call → provider enrichment pipeline. Discovered the MusicBrainz release-group API returns HTTP 400 when `inc=releases+labels+media` is used — `labels` and `media` aren't valid inc params for the release-group endpoint. Fix: changed to `inc=releases`.

4. **Edge analysis test** — Wrote `EdgeAnalysisTest.kt` with 11 structured test categories:
   - Special characters (slashes, diacriticals, parens, umlauts, ampersands)
   - Non-Latin scripts (Japanese, Cyrillic, Korean, Chinese, Hebrew)
   - Garbage/adversarial input (empty, fake, SQL injection, XSS, very long strings)
   - Composite type edge cases (solo artist, classical, many members, new artist, explicit sub-types)
   - Credits edge cases (instrumental, classical, electronic, recent tracks)
   - Genre merge behavior (6 diverse artists across genres)
   - Release editions edge cases (massive reissue catalog, unusual titles, ambiguous names)
   - Provider fallback chains (high-confidence vs obscure albums)
   - Cache behavior (same request twice, measuring speedup)
   - All-types coverage by request kind (ForArtist, ForAlbum, ForTrack)

5. **Analysis** — Reviewed output for patterns: consistent NotFound types, error distributions, merge activation, partial data, date parsing, and type mismatches.

## Findings

### Bugs (need fixing)

#### BUG-1: Genre merge never activates for ForArtist requests

**Severity:** High — the GenreMerger feature from Phase 10 doesn't work for the most common use case.

**Symptoms:** Every ForArtist genre request shows `provider=musicbrainz merged=false` instead of `provider=genre_merger`. Tested with 6 artists (Radiohead, Kendrick Lamar, Miles Davis, Billie Eilish, Amon Amarth, Boards of Canada) — all single-source.

**Root cause:** Last.fm's `enrichGenre()` for ForArtist calls `api.getArtistTopTags()`, which may be returning empty results even though the API key is valid. When only one provider (MusicBrainz) returns Success, `mergeGenreResults()` falls back to returning the single result with its original provider name — the merge path is skipped because `allTags.isEmpty()` triggers the fallback when genreTags are null on the MusicBrainz result.

**Two sub-issues:**
1. Last.fm `getArtistTopTags` may not be returning data for ForArtist requests (investigate API call)
2. MusicBrainz ForArtist genre results may have `genreTags = null` (the `buildGenreTags` returns null when `tagCounts` is empty, but the merge logic falls back to the raw result which doesn't get tagged)

**Where to investigate:**
- `LastFmProvider.kt` line 111-118: `enrichGenre()` ForArtist path
- `LastFmApi.kt`: `getArtistTopTags()` — is it actually calling the API?
- `MusicBrainzMapper.kt` line 142-145: `buildGenreTags()` — does it populate genreTags on artist-level results?
- `DefaultEnrichmentEngine.kt` line 209-238: `mergeGenreResults()` fallback logic

**Note:** Genre merge DID work for Kendrick Lamar in an earlier showcase run (showed `genre_merger` with hip-hop(1.00,[musicbrainz, lastfm])). The discrepancy may be related to rate limiting, request ordering, or ForAlbum vs ForArtist path differences.

#### BUG-2: Foo Fighters timeline has malformed member dates

**Severity:** Medium — timeline events show "03", "07", "10" instead of full years.

**Symptoms:** Foo Fighters timeline member_left events show bare two-digit strings as dates instead of full dates like "2003" or "2003-06-15".

**Root cause:** MusicBrainz returns member relationship dates in varying formats. The `TimelineSynthesizer.extractBandMemberEvents()` may be parsing begin/end dates incorrectly when the date format is non-standard (e.g., just a month "03" instead of "2003-03" or "2003").

**Where to investigate:**
- `TimelineSynthesizer.kt`: `extractBandMemberEvents()` — how does it get dates from BandMember data?
- `BandMember` data class: what fields carry date info? (`activePeriod`? `beginDate`?)
- `MusicBrainzParser.parseBandMembers()`: how are member dates parsed from the API response?

#### BUG-3: ARTIST_TIMELINE returns 0 events for ForAlbum requests

**Severity:** Low-Medium — timeline is primarily a ForArtist feature, but the engine attempts it for ForAlbum too.

**Symptoms:** Abbey Road ForAlbum request returns `ARTIST_TIMELINE` with 0 events, even though The Beatles have extensive timeline data.

**Root cause:** When the request is ForAlbum, identity resolution returns album metadata (not artist metadata). The `synthesizeTimeline()` function needs `beginDate`/`endDate`/`artistType` from the identity result's `Metadata`, but the album identity resolution may not include artist life-span data. The sub-type resolution (ARTIST_DISCOGRAPHY, BAND_MEMBERS) may also fail for ForAlbum because those are ForArtist capabilities.

**Where to investigate:**
- `DefaultEnrichmentEngine.kt`: how composite types handle ForAlbum requests
- Whether ARTIST_TIMELINE should be restricted to ForArtist only (via `IdentifierRequirement` or request type check)

### Fixed during this session

#### FIX-1: RELEASE_EDITIONS HTTP 400 from MusicBrainz (FIXED)

**Was:** `MusicBrainzApi.lookupReleaseGroup()` used `inc=releases+labels+media` — but `labels` and `media` are not valid inc parameters for the release-group endpoint.

**Fix:** Changed to `inc=releases` in `MusicBrainzApi.kt`. Release-group endpoint returns releases with basic fields (title, date, country, barcode). Format/label/catalogNumber are null but editions are otherwise complete.

**Enhancement opportunity:** Use `inc=releases+media` (media IS valid for release-group per MusicBrainz docs — it was `labels` that was invalid). This would populate the `format` field on editions.

### Observations (not bugs, but noteworthy)

1. **Credits coverage is thin** — Only 2/4 test cases returned credits. CREDITS depends on MusicBrainz having recording-level artist-rels, which many recordings lack. Discogs fallback needs `discogsReleaseId` which requires Discogs identity resolution to have matched the album. Classical and very recent tracks are the weakest.

2. **Single-character queries match** — Query `a`/`b` returned SUCCESS from MusicBrainz. Not harmful but worth noting for consumers who might accidentally send minimal input.

3. **Abbey Road ForAlbum request shows wrong snippet for BAND_MEMBERS/CREDITS/etc** — The `resultSummary` display shows Metadata fields (genres, label) for non-Metadata types. This may be a display bug in the test, OR it may mean the identity resolution Metadata is being returned as the result for those types (which would be a real bug in the provider).

4. **ForArtist gets 21/28 types, ForAlbum gets 17/28, ForTrack gets 15/28** — Expected distribution. ForArtist has the most data sources. ForTrack is weakest because many types are album/artist-scoped.

5. **ListenBrainz popularity shows 0 listeners/0 plays for Abbey Road** — The batch popularity endpoint may return zeros for albums (vs individual tracks/artists). Needs investigation on whether the endpoint is being called correctly for ForAlbum.

## Recommended next steps (priority order)

1. **Fix BUG-1 (genre merge)** — Investigate why Last.fm ForArtist genre path isn't contributing. This is the highest-impact issue — the entire GenreMerger feature from Phase 10 is effectively dormant for artist requests.

2. **Fix BUG-2 (timeline dates)** — Trace how BandMember dates flow from MusicBrainz API → parser → TimelineSynthesizer. The date extraction likely needs format normalization.

3. **Investigate BUG-3 (timeline for ForAlbum)** — Decide if ARTIST_TIMELINE should be ForArtist-only, or if the engine should derive artist identity from album requests for composite resolution.

4. **Enhancement: `inc=releases+media` for release-group** — Restore `media` to the inc parameter (it was `labels` that was invalid). This would populate the `format` field on edition results.

5. **Investigate Abbey Road identity result leaking** — Check if BAND_MEMBERS/CREDITS/etc for ForAlbum are returning Metadata instead of their proper types. This would be a routing bug in the provider.

6. **Improve credits coverage** — Consider adding a text-search fallback for credits when no recording MBID is available, or expanding Discogs credits to not require `discogsReleaseId`.
