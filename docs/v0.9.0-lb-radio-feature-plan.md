# Feature Plan: ListenBrainz Radio & Track Preview

> Extends musicmeta's discovery capabilities with a second radio source and a new track preview enrichment type.

---

## Project Rules (CLAUDE.md compliance)

All implementation MUST follow the rules in `CLAUDE.md`. Key constraints for this feature:

### Provider structure
Every provider follows the `Api → Models → Mapper → Provider` pattern. No exceptions.
- `*Api.kt` — raw HTTP calls, returns parsed model DTOs
- `*Models.kt` — data classes for API responses (NOT `@Serializable` — provider models are internal)
- `*Mapper.kt` — stateless `object` that transforms DTOs into `EnrichmentData` types
- `*Provider.kt` — implements `EnrichmentProvider`, orchestrates the above into `EnrichmentResult`

For this feature: LB Radio additions go into existing `ListenBrainzApi.kt`, `ListenBrainzModels.kt`, `ListenBrainzMapper.kt`, `ListenBrainzProvider.kt`. Do NOT create new provider directories. Deezer additions go into existing `DeezerApi.kt`, `DeezerModels.kt`, `DeezerMapper.kt`, `DeezerProvider.kt`.

### Modeling rules
- **Enums for fixed sets** — `radioDiscoveryMode` should be an enum (`RadioDiscoveryMode.EASY/MEDIUM/HARD`), NOT a raw `String`. The plan's earlier sketch used `String` — correct this at implementation time.
- **Sealed classes for variants** — `EnrichmentData.TrackPreview` is a new sealed class variant (data class).
- **Data classes over Pair/Triple/Map** — `ListenBrainzRadioTrack` must be a proper data class, not a Map or JSONObject passed around.
- **`@Serializable` only on public API payload types** — `EnrichmentData.TrackPreview` gets `@Serializable` (it's a public `EnrichmentData` subtype). `ListenBrainzRadioTrack` does NOT (it's an internal provider model).
- **No `!!`** — handle nullability with `?.`, `takeIf`, `?:`, `optString()`, etc.

### Code style
- Files: 200 lines target, 300 max. If `ListenBrainzApi.kt` (currently 164 lines) exceeds 300 after adding `getRadio()`, extract radio-specific code.
- Functions: 20 lines target, 40 max. JSPF parsing should be a separate private function, not inline.
- JSON parsing uses `org.json` (`JSONObject`, `optString()`, `optJSONArray()`) — NOT Gson/Moshi.

### Testing patterns
- Fakes over mocks — use `FakeHttpClient` from `testutil/` for unit tests.
- Test names: backtick style — `` `LB Radio returns RadioPlaylist for known artist` ``
- Given-When-Then with comments:
  ```kotlin
  @Test
  fun `LB Radio returns NotFound when token is missing`() {
      // Given — a provider with no auth token
      // When — enriching ARTIST_RADIO_DISCOVERY
      // Then — returns NotFound, not an error
  }
  ```
- E2E tests gated by `-Dinclude.e2e=true`. LB Radio E2E additionally gated by env var for the token.
- Use `runTest` for unit tests, `runBlocking` for E2E tests (real HTTP).

### Confidence scoring
- Use `ConfidenceCalculator` methods — never hardcode float values. LB Radio results should use `ConfidenceCalculator.authoritative()` (it's the sole provider for that type, and results come from a trusted source with an MBID-based prompt).
- TRACK_PREVIEW should use `ConfidenceCalculator.fuzzyMatch(hasArtistMatch = true)` (same as existing Deezer search-based enrichments).

### Key conventions
- Package: all new code under `com.landofoz.musicmeta` (core module)
- `HttpResult` sealed class for all HTTP calls — never return nullable from API methods when `HttpResult` can express the failure
- `ProviderCapability` registration with explicit `priority` and `identifierRequirement`
- `EnrichmentIdentifiers.withExtra()` for provider-specific IDs (e.g., `"releaseMbid"`, `"artistMbid"`)

## Motivation

`ARTIST_RADIO` is currently single-source (Deezer). This limits discovery in two ways:

1. **Coverage gaps** — Deezer's catalog skews mainstream. Niche/indie artists that ListenBrainz's open community tracks well may return thin or empty Deezer radio results.
2. **No preview playback for unknown tracks** — when radio or discovery results include tracks the user doesn't own, there's no way to hear them. Deezer provides 30-second MP3 previews for most tracks, but musicmeta doesn't expose these.

This plan adds ListenBrainz LB Radio as a second discovery source and `TRACK_PREVIEW` as a new enrichment type, giving consumers the data they need to build discovery UIs with playable previews.

## Design Decisions

### Separate types, not merged

LB Radio and Deezer radio serve **different user intents**:

| | Deezer Radio | LB Radio |
|---|---|---|
| **Algorithm** | Proprietary, editorial-influenced | Community listening data |
| **Ordering** | Curated for listening flow | Ranked by community popularity |
| **Tuning** | None — fixed output | `easy` / `medium` / `hard` modes |
| **Strength** | Mainstream, polished playlists | Niche artists, adventurous picks |
| **Typical use** | "Play more like this" | "Help me discover something new" |

Merging destroys curation intent from both sources. Radio playlists are ordered sequences, not unordered sets like SIMILAR_ARTISTS. A RadioMerger would need to interleave and re-rank, losing the coherence each source provides.

**Decision**: Keep `ARTIST_RADIO` as Deezer. Add `ARTIST_RADIO_DISCOVERY` (or `RADIO_DISCOVERY`) as a new enrichment type backed by ListenBrainz LB Radio.

Consumers choose which flavour to request:
- `ARTIST_RADIO` — "more of what you know" (Deezer)
- `ARTIST_RADIO_DISCOVERY` — "things you haven't heard" (ListenBrainz)
- Both — request both types, present differently in UI

### Track preview as enrichment type

When radio or discovery results include tracks the user doesn't have, the consumer needs a way to let them hear it. Deezer's search API returns a `preview` field (30-second 128kbps MP3 CDN URL) for most tracks.

**Decision**: New `TRACK_PREVIEW` enrichment type. Deezer is the initial provider. Other preview sources could plug in later without changing the consumer API.

```kotlin
val preview = engine.enrich(
    EnrichmentRequest.forTrack("Everything In Its Right Place", "Radiohead"),
    types = setOf(EnrichmentType.TRACK_PREVIEW)
)
val previewUrl = preview.get<EnrichmentData.TrackPreview>(EnrichmentType.TRACK_PREVIEW)?.url
```

This keeps musicmeta's role as "provide data" — the consumer decides when and how to play previews.

### Auth gating per-capability, not per-provider

ListenBrainz currently needs no API key for existing endpoints (popularity, similar artists, discography). LB Radio requires an auth token (free account).

**Decision**: Gate the radio capability only. If `ApiKeyConfig.listenBrainzToken` is present, register the LB Radio provider. If absent, LB Radio is silently unavailable — existing ListenBrainz capabilities continue working without auth. Same pattern as Last.fm/Fanart.tv/Discogs: key present → extra capabilities.

## New Types

### `ARTIST_RADIO_DISCOVERY`

| | |
|---|---|
| **Provider** | ListenBrainz (`GET /1/explore/lb-radio`) |
| **Input** | Artist MBID (from identity resolution). Falls back to artist name if no MBID. |
| **Auth** | ListenBrainz user token via `Authorization: Token {token}` header |
| **Output** | `EnrichmentData.RadioPlaylist` (reuses existing model — see field mapping below) |
| **TTL** | 7 days (`7L * 24 * 60 * 60 * 1000` — matches `ARTIST_RADIO`) |
| **CatalogFilter** | Yes — add to `RECOMMENDATION_TYPES` in `CatalogFilter.kt` |
| **IdentifierRequirement** | `MUSICBRAINZ_ID` (preferred) or `NONE` (falls back to name prompt) |
| **Priority** | 100 (sole provider for this type) |

**API call**:
```
GET /1/explore/lb-radio?prompt=artist:(a74b1b7f-71a5-4011-9441-d0b5e4122711)&mode=easy
Authorization: Token {listenBrainzToken}
```

When no MBID is available, use name: `prompt=artist:(Radiohead)`.

**Response** (JSPF playlist):
```json
{
  "payload": {
    "jspf": {
      "playlist": {
        "track": [
          {
            "title": "Track Name",
            "creator": "Artist Name",
            "identifier": ["https://musicbrainz.org/recording/{recording_mbid}"],
            "duration": 234000,
            "album": "Album Name",
            "extension": {
              "https://musicbrainz.org/doc/jspf#playlist": {
                "artist_identifiers": ["https://musicbrainz.org/artist/{artist_mbid}"],
                "release_identifier": "https://musicbrainz.org/release/{release_mbid}"
              }
            }
          }
        ]
      }
    }
  }
}
```

**Mapping to existing `RadioTrack` model** (verified — all fields are compatible):

| JSPF field | `RadioTrack` field | Notes |
|---|---|---|
| `title` | `title: String` | Direct mapping |
| `creator` | `artist: String` | Direct mapping |
| `album` | `album: String?` | Direct mapping — `RadioTrack.album` exists and is nullable |
| `duration` | `durationMs: Long?` | Direct mapping — JSPF duration is already in ms |
| `identifier[0]` | `identifiers.musicBrainzId` | Extract UUID from URL: `"https://musicbrainz.org/recording/{uuid}"` → strip prefix |
| `extension..artist_identifiers[0]` | `identifiers.extra["artistMbid"]` | Extract UUID from URL, store via `withExtra()` |
| `extension..release_identifier` | `identifiers.extra["releaseMbid"]` | Extract UUID from URL, store via `withExtra()` |

**MBID extraction helper**: Parse `https://musicbrainz.org/{type}/{uuid}` URLs → extract the UUID suffix. Shared utility since all three JSPF identifier fields use this format. Use `substringAfterLast("/")`.

**JSPF extension namespace**: The MusicBrainz JSPF extension URI is `https://musicbrainz.org/doc/jspf#playlist`. Access via `track.optJSONObject("extension")?.optJSONObject("https://musicbrainz.org/doc/jspf#playlist")`. If the extension key varies, fall back to iterating extension keys and picking the first object that has `artist_identifiers`.

**Mode configuration**: Add `RadioDiscoveryMode` enum and `radioDiscoveryMode: RadioDiscoveryMode` to `EnrichmentConfig`. Thread to the API call via `ListenBrainzProvider` constructor receiving the config, same pattern as `radioLimit` for Deezer radio.

```kotlin
// New enum — fixed set, never string constants (per CLAUDE.md modeling rules)
enum class RadioDiscoveryMode(val apiValue: String) {
    EASY("easy"),
    MEDIUM("medium"),
    HARD("hard"),
}

// In EnrichmentConfig — add alongside existing radioLimit
data class EnrichmentConfig(
    // ... existing fields ...
    val radioDiscoveryMode: RadioDiscoveryMode = RadioDiscoveryMode.EASY,
)
```

**Auth header threading**: `ListenBrainzApi` currently has no auth support. Add optional `authToken: String?` constructor param. When present, add `Authorization: Token $authToken` header to requests that need it. The `getRadio()` method requires it; existing methods don't.

```kotlin
class ListenBrainzApi(
    private val httpClient: HttpClient,
    private val rateLimiter: RateLimiter,
    private val authToken: String? = null,  // new — only needed for LB Radio
)
```

### `TRACK_PREVIEW`

> **Why this is a standalone type, not embedded in radio responses**: `TRACK_PREVIEW` is context-agnostic — it works for any track regardless of how the consumer discovered it. A radio track the user doesn't own, a similar artist's top track, a standout track from a similar album, a search result — all use the same `enrich(forTrack(...), TRACK_PREVIEW)` call. Embedding preview URLs inside radio or similar-artist responses would limit previews to those contexts and duplicate the resolution logic. As a standalone type, one feature serves every discovery surface the consumer builds.

| | |
|---|---|
| **Provider** | Deezer (`GET /search/track`) |
| **Input** | Track title + artist name (from `ForTrack` request) |
| **Auth** | None |
| **Output** | `EnrichmentData.TrackPreview` (new sealed class variant) |
| **TTL** | 24 hours (`1L * 24 * 60 * 60 * 1000` — preview URLs are CDN links that may rotate) |
| **CatalogFilter** | No — do NOT add to `RECOMMENDATION_TYPES`. Previews are specifically for tracks the user *doesn't* have. |
| **IdentifierRequirement** | `NONE` — uses title + artist text search, no MBID needed |
| **Priority** | 100 (sole provider for this type) |

**New data model** (add to `EnrichmentData` sealed class in `EnrichmentData.kt`):
```kotlin
@Serializable
data class TrackPreview(
    val url: String,              // 30s MP3 preview URL
    val durationMs: Long = 30000, // always 30s for Deezer
    val source: String,           // "deezer"
) : EnrichmentData()
```

**Existing code that needs modification** — `DeezerTrackSearchResult` currently discards the `preview` field:

```kotlin
// Current DeezerModels.kt — missing preview and duration
data class DeezerTrackSearchResult(
    val id: Long,
    val title: String,
    val artistName: String,
)

// Updated — add preview URL and duration
data class DeezerTrackSearchResult(
    val id: Long,
    val title: String,
    val artistName: String,
    val previewUrl: String? = null,  // new
    val durationSec: Int? = null,    // new
    val albumTitle: String? = null,  // new — useful context
)
```

**`DeezerApi.searchTrack()` modification** — currently at `DeezerApi.kt:136`, returns first match without extracting `preview`. Update the parsing block:

```kotlin
// Current (line ~151)
return DeezerTrackSearchResult(
    id = track.optLong("id"),
    title = track.optString("title", ""),
    artistName = artistName,
)

// Updated — extract preview URL from same response
return DeezerTrackSearchResult(
    id = track.optLong("id"),
    title = track.optString("title", ""),
    artistName = artistName,
    previewUrl = track.optString("preview").takeIf { it.isNotBlank() },
    durationSec = track.optInt("duration").takeIf { it > 0 },
    albumTitle = track.optJSONObject("album")?.optString("title"),
)
```

The `preview` field is already in every Deezer track search response — we're just not parsing it. Zero extra API calls.

**ArtistMatcher validation**: The existing `enrichSimilarTracks()` already validates `ArtistMatcher.isMatch()` on search results before using them. `TRACK_PREVIEW` should follow the same pattern — verify the search result's artist matches before returning the preview URL.

## Consumer Usage Patterns

### Pattern 1: Discovery radio with library-first playback (Cascade use case)

```kotlin
// 1. Get discovery radio (catalog-filtered: library tracks sorted first)
val profile = engine.artistProfile("Radiohead",
    types = setOf(ARTIST_RADIO_DISCOVERY)
)

// 2. For each track in the radio playlist
for (track in profile.radioDiscovery?.tracks.orEmpty()) {
    if (catalogProvider.isAvailable(track)) {
        // Play from local library at full quality
    } else {
        // Lazy-resolve preview
        val preview = engine.enrich(
            EnrichmentRequest.forTrack(track.title, track.artist),
            types = setOf(TRACK_PREVIEW)
        )
        if (preview.trackPreview()?.url != null) {
            // Show with "30s preview" indicator, play on tap
        } else {
            // Show as "not available" — no preview exists
        }
    }
}
```

## Risks & Mitigations

### Handled by musicmeta (infrastructure, not opinion)

| Risk | Impact | How musicmeta handles it |
|------|--------|--------------------------|
| **Preview URL expiration** | Low — CDN URLs are stable for days/weeks but not permanent | `TRACK_PREVIEW` gets a short TTL (24h). When a consumer calls `enrich()` after expiry, the cache misses and a fresh URL is resolved automatically. The consumer never thinks about expiration — it's just correct caching behavior, same as any other enrichment type. |
| **Niche track match rate** | Medium — LB Radio's strength (niche artists) is Deezer's weakness (thin catalog) | Already handled by existing patterns. `DeezerApi.searchTrack()` + `ArtistMatcher.isMatch()` returns `NotFound` when there's no confident match — never a wrong track's preview. Consumer gets `NotFound` and shows "no preview available" state. No special handling needed. |
| **Rate limiting on batch preview resolution** | Medium — 25 tracks × 1 search each = 25s through rate limiter | `enrichBatch()` already exists and handles throttling internally via the per-provider rate limiter. Consumers use the existing batch API — they don't need to know about Deezer's rate limits or implement their own throttling. The rate limiter is infrastructure the library already provides. |

### Handled by consumer (policy, not plumbing)

| Risk | Impact | Why musicmeta should not handle it |
|------|--------|-------------------------------------|
| **Deezer ToS for commercial use** | Medium — free for non-commercial, commercial needs Deezer agreement | Enforcing usage policies is the consumer's responsibility. musicmeta is a data pipe — adding usage counters, commercial-use flags, or rate caps beyond API limits would be opinionated. Document "Deezer previews are free for non-commercial use, check their terms for commercial" in the provider guide. That's the right level. |

### Other risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| LB Radio requires auth — raises the bar for "works without API keys" | Medium — 8/11 providers still keyless; LB Radio is opt-in | Gate per-capability. Provider docs clearly state which features need auth. |
| LB Radio prompt syntax changes | Low — it's a stable public API | Pin to known prompt format. Version the prompt builder if syntax evolves. |

## What This Enables

For a self-hosted music player like Cascade:
- **Library-first radio** — ARTIST_RADIO filtered by catalog, play local tracks at full quality
- **Discovery with previews** — ARTIST_RADIO_DISCOVERY surfaces unknown tracks, TRACK_PREVIEW gives 30-second tasters
- **Depth control** — easy/medium/hard modes let the user choose how adventurous to be
- **Buy/add signal** — tracks the user previews and likes become candidates for library acquisition

For musicmeta as a library:
- ARTIST_RADIO stays single-source, simple, no merger complexity
- New type is cleanly additive — no breaking changes
- TRACK_PREVIEW is independently useful beyond radio (any track lookup can get a preview)
- "We provide the pipe and plumbing — consumers decide how to use it"
